# 计量选型 MVP 开发计划

## 1. 项目目标

开发一个最小可用的前后端分离项目，用于计量工具选型。

项目要求：

1. 前端只有一个页面。
2. 不需要登录权限。
3. 不需要管理后台。
4. 数据源来自钉钉多维表。
5. 后端不直接接数据库，先采用本地 JSON 文件作为查询数据源。
6. 数据更新方式采用定时同步钉钉到本地 JSON 文件。
7. 前端保留现有计量工具选型页面中的联动、测量范围解析、MPE 计算和结果展示逻辑。

本项目的核心原则：

1. 优先复用现有代码。
2. 优先降低实现复杂度。
3. 先跑通完整链路，再优化结构。

---

## 2. 最终交付物

AI 最终需要交付以下内容：

1. 一个后端服务项目。
2. 一个钉钉同步脚本。
3. 一个本地 JSON 数据文件生成逻辑。
4. 一个可运行的单页前端页面。
5. 前后端联调完成后的可运行版本。
6. 一个简短的运行说明文档。

---

## 3. 范围与非范围

### 3.1 本次必须实现

1. 从钉钉分页拉取记录。
2. 将钉钉记录清洗成统一结构。
3. 保存为本地 JSON 文件。
4. 后端提供筛选项接口。
5. 后端提供查询接口。
6. 前端从接口加载数据并展示结果。
7. 前端保留现有测量要求区间解析和 MPE 判断逻辑。
8. 支持一级分类、二级分类、名称、型号、厂家等筛选。
9. 支持定时同步。

### 3.2 本次不要实现

1. 登录权限。
2. 用户系统。
3. 管理后台。
4. 数据库设计与迁移。
5. Docker 化部署。
6. 复杂权限控制。
7. 查询历史记录。
8. 审批流。
9. 多页面应用。

## 4. 推荐技术栈

为了让 AI 更容易一次性完成，技术栈固定如下：

### 4.1 后端

使用 Python + FastAPI。

原因：

1. 返回 JSON 简单直接。
2. 接口定义清晰。
3. 适合快速开发。
4. 对后续扩展 SQLite 或数据库也比较顺手。

### 4.2 前端

继续使用现有 HTML + 原生 JavaScript，不引入前端框架。

原因：

1. 当前已有可复用页面。
2. 不需要额外搭建 Vue 或 React 工程。
3. 更适合 MVP 快速落地。

### 4.3 数据存储

使用本地 JSON 文件。

原因：

1. 实现成本最低。
2. 500+ 条数据查询压力很小。
3. 后续如果需要可以升级为 SQLite。

### 4.4 定时任务

Windows 环境下使用任务计划程序。

## 6. 数据流设计

系统数据流如下：

1. 同步脚本调用钉钉接口。
2. 获取 access_token。
3. 分页拉取多维表记录。
4. 提取前端需要的字段。
5. 统一清洗为标准结构。
6. 保存到本地 JSON 文件。
7. 前端打开页面时调用后端接口获取筛选项。
8. 用户填写筛选条件后调用查询接口。
9. 后端从本地 JSON 中读取并按条件过滤。
10. 前端拿到结果后继续执行现有的测量区间解析、量程判断、MPE 计算、结果渲染逻辑。

## 7. 标准 JSON 数据结构

AI 必须先定义统一的数据结构，后端接口和前端页面都依赖这个结构。

### 7.1 单条工具记录结构

```json
{
  "record_id": "rec_xxx",
  "一级分类": "长度",
  "二级分类": "卡尺",
  "名称": "游标卡尺",
  "型号": "0-150mm",
  "生产厂家": "某厂家",
  "测量下限": "0mm",
  "测量上限": "150mm",
  "准确度": "±0.02mm",
  "最大允许误差": "±0.02mm",
  "分度值": "0.01mm",
  "应用场景": "长度测量",
  "图片": "https://example.com/image.png",
  "原始字段": {}
}
```

### 7.2 字段约束

1. 所有字段统一使用字符串或空字符串，避免前端处理时出现类型混乱。
2. 如果原始记录缺失字段，则补空字符串。
3. `原始字段` 可选保留，用于调试；MVP 可保留也可不保留。
4. 输出字段名尽量保持和当前 HTML 使用的字段一致，减少前端改造量。

### 7.3 本地 JSON 文件整体结构

建议如下：

## 8. 后端接口设计

AI 必须严格按以下接口契约开发。

### 8.1 健康检查接口

#### 路径

`GET /api/health`

#### 返回示例

```json
{
  "success": true,
  "message": "ok"
}
```

### 8.2 获取筛选项接口

#### 路径

`GET /api/options`

#### 功能

从本地 JSON 中提取前端页面需要的筛选项。

#### 返回示例

```json
{
  "success": true,
  "data": {
    "categories": ["长度", "温度", "压力"],
    "sub_categories": ["卡尺", "千分尺"],
    "equipment_names": ["游标卡尺", "数显卡尺"],
    "manufacturers": ["厂家A", "厂家B"]
  }
}
```

#### 说明

1. 这里返回全局筛选项即可。
2. 前端也可以根据查询结果再做局部联动。
3. MVP 阶段不要把接口拆得太细。

### 8.3 查询接口

#### 路径

`POST /api/search`

#### 请求体示例

```json
{
  "category": "长度",
  "sub_category": "卡尺",
  "equipment_name": "游标卡尺",
  "model": "",
  "manufacturer": "",
  "keyword": ""
}
```

#### 查询规则

1. 仅按结构化字段进行过滤。
2. `measurement_requirement` 不在后端做判断。
3. 测量范围解析和 MPE 计算留给前端复用现有逻辑。
4. 空字段表示不作为过滤条件。

#### 返回示例

```json
{
  "success": true,
  "data": {
    "total": 12,
    "records": [
      {
        "record_id": "rec_xxx",
        "一级分类": "长度",
        "二级分类": "卡尺",
        "名称": "游标卡尺",
        "型号": "0-150mm",
        "生产厂家": "某厂家",
        "测量下限": "0mm",
        "测量上限": "150mm",
        "准确度": "±0.02mm",
        "最大允许误差": "±0.02mm",
        "分度值": "0.01mm",
        "应用场景": "长度测量",
        "图片": ""
      }
    ]
  }
}
```

### 8.4 错误返回格式

统一格式如下：

```json
{
  "success": false,
  "message": "错误信息"
}
```

## 9. 钉钉同步脚本要求

AI 编写同步脚本时，必须满足以下要求。

### 9.1 输入

通过配置文件或环境变量读取：

1. 钉钉 AppKey
2. 钉钉 AppSecret
3. 多维表 appToken
4. tableId

### 9.2 处理流程

1. 获取 access_token。
2. 分页拉取全部记录。
3. 提取项目所需字段。
4. 清洗字段为空值、对象值、枚举值。
5. 统一转换为标准 JSON 结构。
6. 保存到 `backend/data/tools.json`。
7. 保存前先写临时文件，再原子替换，避免写入中断导致 JSON 损坏。

### 9.3 失败处理

1. 如果同步失败，不覆盖旧文件。
2. 输出清晰日志。
3. 返回非 0 退出码，方便任务计划程序识别失败。

### 9.4 可复用要求

尽量复用现有 [dingtalk_list_records_paged.py](dingtalk_list_records_paged.py) 中的分页与字段读取逻辑，不要从零重写一套不同风格的实现。

AI 改造前端时，以 [计量工具选型.html](计量工具选型.html) 为基础，不要重做视觉和业务逻辑。

### 10.1 保留内容

必须保留：

5. 设备量程校验逻辑。
6. MPE 计算逻辑。
7. 结果卡片展示逻辑。

### 10.2 替换内容

把原来模板注入方式：


替换为接口驱动方式：

## 11. 分阶段开发任务

下面的任务要按顺序执行，AI 一次完成一个阶段，不要把全部工作混在一起。

### 阶段 1：搭建后端骨架

目标：先让后端服务跑起来。

任务：

1. 创建 FastAPI 项目结构。
2. 实现 `/api/health`。
3. 配置基础启动方式。
4. 提供 `requirements.txt`。

完成标准：

1. 服务可启动。
2. 访问 `/api/health` 返回成功 JSON。

### 阶段 2：实现钉钉同步脚本

目标：先生成本地 JSON 文件。

任务：

1. 从现有钉钉脚本提取通用函数。
2. 实现 access_token 获取。
3. 实现分页拉取。
4. 实现字段清洗。
5. 生成 `tools.json`。

完成标准：

1. 脚本手动执行后生成 JSON。
2. JSON 中有 `updated_at`、`total`、`records`。
3. 记录字段结构统一。

### 阶段 3：实现 options 和 search 接口

目标：先让后端具备查询能力。

任务：

1. 实现 JSON 文件读取逻辑。
2. 实现筛选项提取逻辑。
3. 实现查询过滤逻辑。
4. 统一返回格式。

完成标准：

1. `/api/options` 可返回筛选项。
2. `/api/search` 可按条件返回记录。
3. 文件不存在时有明确错误返回。

### 阶段 4：前端接入接口

目标：让页面脱离模板注入，改为请求后端接口。

任务：

1. 复制现有 HTML 到前端页面。
2. 删除 Dify 模板变量依赖。
3. 页面加载时调用 `/api/options`。
4. 查询时调用 `/api/search`。
5. 保留现有测量范围和 MPE 逻辑。

完成标准：

1. 页面能正常打开。
2. 筛选项可正常加载。
3. 查询结果可正常展示。

### 阶段 5：异常处理与验收

目标：让项目达到 MVP 可用状态。

任务：

1. 增加空结果提示。
2. 增加接口异常提示。
3. 增加同步失败不覆盖旧文件逻辑。
4. 增加简单运行说明。

完成标准：

1. 无数据时页面不报错。
2. 后端报错时前端有可理解提示。
3. 项目可按文档启动运行。

## 17. 备注
